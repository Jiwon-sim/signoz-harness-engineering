# ECC — Engineering Compliance Check (상시 규칙)

> 코드 변경의 품질 절차를 정의하는 7단계 기준. 강제(하드 차단)가 아니라 **보고상 게이트**로 운영한다.
> 검증은 가능하면 실제 수행하고, 실패는 실패로 정직하게 보고한다. (하네스 원칙: `CLAUDE.md`)
> 실행은 `/ecc <작업>` 명령으로, `code-reviewer` 는 5·6단계의 출처로 이 문서를 참조한다.

## 7단계
| # | 단계 | 핵심 |
|---|------|------|
| 1 | 계획 수립 | 영향 범위·접근 방식 명시 |
| 2 | 영향 분석 | 의존성·호출처·에디션(Community/EE) 판별 (→ `/analyze-feature`) |
| 3 | 구현 | implementer 에이전트 위임, 한 응답 ≤5파일 |
| 4 | 빌드 검증 | `make go-build-community` / `pnpm build` (가능하면 실행) |
| 5 | 보안 검토 | 입력검증·권한·비밀정보·주입·멀티테넌시 격리 → `code-reviewer` 에이전트 |
| 6 | 코드 리뷰 | 정확성·기술 품질 → `/code-review`(빌트인) → 사람 승인 |
| 7 | 데드코드 정리 | 미사용 import/export/변수·디버그 로그 제거 |

> **5·6단계 역할 분리** (겹침 방지): 보안 검토(5)는 `code-reviewer` 에이전트가 전담하고,
> 일반 코드 리뷰(6)는 빌트인 `/code-review`를 사용한다. `code-reviewer`로 6단계를 중복 수행하지 않는다.

## 복잡도별 적용 (단계 선택)
- **오타·주석·한 줄 수정** → 1, 3 단계만.
- **일반 기능** → 전 단계(1~7).
- **API·DB·보안 변경** → 전 단계 + **사람 승인 필수**(6단계 후).

## 운영 원칙
- 각 단계 통과 여부를 체크박스로 보고한다.
- 미통과 사항은 **문제와 원인을 정직하게 보고**하고, 필요하면 사람 판단으로 다음 단계 진행 여부를 정한다.
- **하드 차단하지 않는다** — "실패 시 다음 단계 진입 금지" 식의 강제 게이트를 만들지 않는다.
